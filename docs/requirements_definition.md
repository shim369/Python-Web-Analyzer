# サイトリニューアル可否自動判定ツール 要件定義書

## 1. システム概要

本システム（英名: `web_analyzer`）は、既存の古いWebサイト（純粋なドメイン名リスト）からスクレイピング等によって各種情報を自動収集し、新しいテンプレートフォーマットへの移行（リニューアル）が容易かどうかを自動判定・可視化するための業務効率化ツールである。

ユーザーは、画面上で「担当者名」および「判定基準となるページ数（しきい値）」を入力・設定し、対象ドメインの記述されたExcelファイルをインポートする。システムは裏側で非同期に各サイトの調査を実行し、判定・解析結果を付加したExcelファイルを生成、ユーザーがダウンロードできるようにする。

---

## 2. 機能要件

### 2.1. 画面入力・パラメータ設定機能

システム実行時に、ユーザーは画面上のフォームから以下のパラメータを指定できるものとする。

* **担当者名（必須）**: ツールを実行するオペレーターの名前（文字列）。
* **判定しきい値（数値入力、デフォルト値あり）**:
* **閾値1（デフォルト: 10）**: これ以下かつCMSなしの場合に「◎」と判定するページ数。
* **閾値2（デフォルト: 15）**: これ以下の場合に「◯」と判定するページ数。
* **閾値3（デフォルト: 20）**: これ以下の場合に「△」と判定するページ数。これを超えると「×」判定となり、不可の理由に対象文言が出力される。



### 2.2. ファイル入出力機能

* **Excelインポート**:
* `A列（サイト名）`に**純粋なドメイン名**（例: `sohtani.pythonanywhere.com`）が記載されたExcelファイル（`.xlsx`）を受け取る。
* システムは解析時、自動的に `https://` などのプロトコルを補完してリクエストを行う。


* **Excelエクスポート**:
* 解析完了後、元の行順（A列の並び）を完全に維持したまま、B列〜N列に解析・判定結果を書き込んだExcelファイルを出力する。



### 2.3. 各列のデータ取得・出力ロジック仕様

インポートされたExcelに対し、システムは以下の仕様に基づき各列のデータを生成する。

| 列 | 項目名 | データ型 | 処理・取得・判定ロジック |
| --- | --- | --- | --- |
| **A** | サイト名 | 文字列 | インポートされたドメイン名（そのまま維持）。 |
| **B** | 調査結果 | 文字列 | 画面指定の閾値に基づき判定。(1) `I列（使用CMS）`が空欄、かつ `H列` ≦ **【閾値1】** ➔ **◎** ／ (2) 上記以外、かつ `H列` ≦ **【閾値2】** ➔ **◯** ／ (3) 上記以外、かつ `H列` ≦ **【閾値3】** ➔ **△** ／ (4) いずれにも該当しない（**【閾値3】**を超える）場合 ➔ **×** |
| **C** | SSLあり | 文字列 | トップページに `https://` でアクセスを試み、有効なSSL証明書が確認できれば **◯**、不可なら **×** または空欄。 |
| **D** | SSL常時 | 文字列 | 下層ページ等でもSSLが有効、かつ `http://` へのアクセスが `https://` へ自動リダイレクトされる等、サイト全体で常時SSL化されていれば **◯**、そうでなければ **×**。 |
| **E** | 階層数 | 整数 | サイト内全体の最大階層数を表示。URLのディレクトリ構造（スラッシュ `/` の数）を基準とする。（例: `https://domain/category/python/` ➔ **2** とみなす） |
| **F** | svcmd | 文字列 | 常に**空欄**（NULL値）として出力する。 |
| **G** | 構成 | 文字列 | サイトのHTMLからグローバルナビゲーションを検出。そのメニューテキストを**改行区切りの縦並び**で表示。見つからなければ空欄。 |
| **H** | ページ数 | 整数 | 同一ドメイン内のすべてのリンクを巡回し、総ページ数を集計。ただし、**10秒以内**に取得（タイムアウト）できなければ、処理を打ち切り**空欄**とする。 |
| **I** | 使用CMS | 文字列 | メタタグ、ディレクトリ構造、レスポンスヘッダー等からCMS（WordPressなど）の利用を識別し表示。なければ空欄。 |
| **J** | 用途 | 文字列 | トップページのメタタグ `<meta name="description" content="...">` の文字列をそのまま抽出して表示。 |
| **K** | 問合せ項目 | 文字列 | 問い合わせページ（URLに `contact` 等を含む）を探索。制限は設けないが、**すぐに見つからない、あるいはフォームの解析ができない場合は空欄**とする。発見時は項目名を**改行区切りの縦並び**で表示。 |
| **L** | 不可の理由 | 文字列 | 条件に合致した場合に理由を出力（複数合致時は併記）。 (1) `H列（ページ数）` ≧ **【閾値3】** ➔ 「**ページ数が多いため**」 ／ (2) サイト内に会員ログイン機能（`login`リンク等）を検出 ➔ 「**会員ログイン機能があるため**」 ※該当なしは空欄。 |
| **M** | 備考 | 文字列 | サイト解析中に特殊な特徴（外部サービスの埋め込み、特殊なスクリプト等）を検知した場合にテキストを出力。基本は空欄。 |
| **N** | 担当 | 文字列 | **アプリ実行時にユーザーが画面上で手動入力した「担当者名」**をそのまま全行に出力する。 |

## 3. 非機能要件

### 3.1. 性能・制限事項（スクレイピング制御）

* **タイムアウト管理**: 1ドメインあたりのクロールおよびページ数集計処理（H列）は最大10秒でタイムアウトとする。これにより、特定の大規模サイトや応答の遅いサイトによってツール全体の処理がスタックするのを防ぐ。
* **クローリングの安全性**: 対象サーバーへ過度な負荷を与えないよう、同一ドメイン内を巡回する際は適切なディレイ（間隔）を設けてリクエストを送信する。
* **Contactページの早期離脱**: 複雑なサイト構造において問い合わせページが発見できない場合、無限ループや過度なネスト探索を回避するため、一定の探索ステップで見つからなければ即座にスキップ（空欄）とする。

### 3.2. 処理方式

* **非同期キュー処理**: 複数ドメインが記載されたExcelファイルがアップロードされた場合、Webアプリケーションのメインプロセスとは別に、バックグラウンド（非同期スレッドやタスクキュー）で安全にスクレイピングを実行する。
* **進捗の可視化**: 処理中は画面側へ現在の処理件数（例: `25 / 100件 完了`）をリアルタイム、または定期的な自動更新（ポーリング）によって通知する。
